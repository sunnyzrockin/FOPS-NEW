## 1. How financial fields are captured today

### 1a) Fixed columns on `shift_reports`
All `DECIMAL(10,2) DEFAULT 0`, all **free-entry** (no formula enforcement):

| Column            | Semantic meaning today (inferred)             |
| ----------------- | --------------------------------------------- |
| `total_sales`     | Sum of fuel + shop (or whatever the staff types) |
| `fuel_sales`      | Total fuel revenue ($)                        |
| `shop_sales`      | Total shop / convenience revenue ($)          |
| `total_litres`    | Total litres sold across all grades           |
| `total_revenue`   | "Headline" revenue — meant to be the truth     |
| `eftpos`, `motorpass`, `cash`, `accounts` | Payment-method breakdown ($)     |
| `beverages`, `hot_food` | Shop sub-categories ($)                 |
| `drive_offs`      | Drive-off loss ($)                            |
| `dips`            | Dip-related $ adjustment (poorly named)       |
| `difference_value`| Stored variance value (rarely populated)      |

### 1b) Dynamic fields (per-site)
`site_field_configs` (category = `sales`) lets each site add custom inputs
(e.g. "ULP Litres", "Diesel Litres", "Premium Sales"). At submit time
(`/app/lib/api/handlers/reports.js`):

- If the field's `key` exactly matches one of the fixed allow-list columns, it
  writes to that column.
- Otherwise, it writes to `custom_values` JSONB.

**No site-level total/component reconciliation runs at submit.**

### 1c) Banking is separate from revenue
`eftpos + motorpass + cash + accounts` is a **payment-method breakdown** of
cash received, not a revenue figure. The dashboard already correctly labels
this as `totalBanking` — that part is fine.

## 2. How each dashboard aggregates the same concept

### Field: "Revenue"

| Endpoint / file                                          | Formula |
| -------------------------------------------------------- | ------- |
| `/api/dashboard/stats` (`dashboard.js:241`)              | `total_revenue || total_sales` |
| `/api/dashboard/revenue-chart` (`dashboard.js:472`)      | `total_sales` (only!) |
| `/api/dashboard/12-month-trend` (`executive-dashboard.js:78`) | `total_revenue || total_sales` |
| `/api/dashboard/variance` (`executive-dashboard.js:134`) | `total_revenue || total_sales` |
| `/api/dashboard/top-performers` (`executive-dashboard.js:205`) | `total_revenue || total_sales` |
| `/api/dashboard/timeseries` metric=revenue (`timeseries.js:40`) | `total_revenue || total_sales` |
| `daily_rollups` (`dashboard.js:114-129`)                 | `total_sales` (only!) |

🔴 **Inconsistency: revenue-chart and daily rollups use `total_sales` only, while every other endpoint falls back through `total_revenue` first.** A site that fills in only `total_revenue` (and leaves `total_sales=0`) will appear in the KPI cards but be invisible on the daily chart.

### Field: "Fuel Sales"
Every endpoint uses `r.fuel_sales` directly. No aggregation from per-grade fields. **If a site uses dynamic fields like `ulp_sales` + `diesel_sales` in `custom_values`, those amounts are never folded into `fuel_sales` — they're silently dropped from the KPI.**

### Field: "Total Litres"
Same pattern as fuel_sales — `r.total_litres` only. Per-grade litres in `custom_values` (`ulp_litres`, `diesel_litres`, etc.) are only used by the *volume-by-grade* chart, never folded back into `total_litres` totals.

### Field: "Banking"
`eftpos + motorpass + cash + accounts`. Consistent everywhere. ✅

## 4. Root causes

1. **No canonical formula module.** Each endpoint re-implements `revenue` with its own `||` fallback.
2. **No validation at entry.** Staff can submit `fuel=5000, shop=1200, total_sales=6200, total_revenue=0` and the system accepts it.
3. **Two parallel "totals":** `total_sales` and `total_revenue` are both stored, and the rules for which one "wins" are different per endpoint.
4. **Dynamic fields are decoupled from totals.** A site with `ulp_litres + diesel_litres` in `custom_values` will have `total_litres = 0` and nobody catches it.

## 5. Proposed canonical financial model (FOR YOUR SIGN-OFF)

> Once you agree to this, **`/app/lib/financials.js` becomes the single source of truth**. Every dashboard, export, and PDF will route through it. The brief's STEP 2.

### 5.1 Canonical formulas

```js
// All amounts in dollars. Litres in litres. All inputs read from a normalised
// shift_report row that has fixed columns + custom_values merged.

shop_sales      = report.shop_sales                                            // as entered
fuel_sales      = report.fuel_sales || sum(per-grade $ in custom_values)       // PRIMARY: fixed column. FALLBACK: sum of per-grade dollar fields.
other_income    = sum of custom_values numeric keys tagged 'other_income'      // future: optional category
total_sales     = fuel_sales + shop_sales + other_income                       // DERIVED — never trust the typed-in total
total_revenue   = total_sales                                                   // collapse the two — see §5.2
banking         = eftpos + motorpass + cash + accounts                         // unchanged (already consistent)
total_litres    = report.total_litres || sum(per-grade litres in custom_values) // PRIMARY: fixed column. FALLBACK: sum of per-grade litre fields.

// Reconciliation flag (stored on the row)
reconciles      = abs((report.total_sales || total_sales) − total_sales) ≤ 0.50
                  AND
                  abs((report.total_revenue || total_sales) − total_sales) ≤ 0.50
                  AND
                  abs(banking − total_sales) ≤ 25                              // small float on cash-handling
```

### 5.2 Big call: **collapse `total_revenue` into `total_sales`**

There is no business meaning for them to differ. Today it's only a source of confusion. **`total_revenue` becomes a derived alias that always equals `total_sales`** at read time. The column stays in the DB (we don't drop columns this sprint) but writes to it are ignored by the canonical layer.

### 5.3 What gets enforced at entry (STEP 3)

On `POST /api/reports` and `PATCH /api/reports/:id`:

1. Compute `total_sales` canonically from components.
2. If the submitter typed in a different `total_sales` (or `total_revenue`), **overwrite it** with the canonical value — and store the submitter's value in `custom_values._submitted_totals` for audit.
3. If `|banking − total_sales| > $25`, set `reconciles=false` + `reconciliation_reason='banking_mismatch'`. **Warn, don't block** (per brief default).

### 5.4 Schema additions (additive, no breaks)

```sql
ALTER TABLE shift_reports
  ADD COLUMN IF NOT EXISTS reconciles BOOLEAN DEFAULT TRUE,
  ADD COLUMN IF NOT EXISTS reconciliation_reason TEXT,
  ADD COLUMN IF NOT EXISTS canonical_total_sales DECIMAL(10,2),  -- cached for fast queries
  ADD COLUMN IF NOT EXISTS canonical_total_litres DECIMAL(10,2);

CREATE INDEX IF NOT EXISTS idx_shift_reports_reconciles
  ON shift_reports (site_id, date) WHERE reconciles = FALSE;
```

### 5.5 Data Integrity view

A new **Owner-only "Data Integrity"** tab listing reports with `reconciles=false`, grouped by site, with one-click jump to the report editor.
